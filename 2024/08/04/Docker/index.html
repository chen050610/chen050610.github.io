Docker

  • Linux 容器不是模拟一个完整的操作系统而是对进程进行隔离。有了容器，就可以将软件运行所需的所有资源打包到一个隔离的容器中。容器与虚拟机不同，不需要捆绑一整套操作系统，只需要软件工作所需的库资源和设置。系统因此而变得高效轻量并保证部署在任何环境中的软件都能始终如一地运行

  • 传统虚拟机技术是虚拟出一套硬件后，在其上运行一个完整操作系统，在该系统上再运行所需应用进程;

  • 容器内的应用进程直接运行于宿主的内核，容器内没有自己的内核且也没有进行硬件虚拟。因此容器要比传统虚拟机更为轻便。

  • 每个容器之间互相隔离，每个容器有自己的文件系统，容器之间进程不会相互影响，能区分计算资源。

  • Docker是内核级虚拟化，其不像传统的虚拟化技术一样需要额外的Hypervisor支持，所以在一台物理机上可以运行很多个容器实例，可大大提升物理服务器的CPU和内存的利用率。

  • Docker的安装和仓库

Docker官网

Docker镜像仓库

  • Docker 并非是一个通用的容器工具，它依赖于已存在并运行的 Linux 内核环境。
    一个隔离的文件环境，因此它执行的效率几乎等同于所部署的 Linux 主机。Docker 实质上是在已经运行的 Linux 下制造了因此Docker 必须部署在 Linux 内核的系统上
    如果其他系统想部署 Docker 就必须安装一个虛拟 Linux 环境。

  • 虚拟机的内核版本应该大于7版本的

  • docker为什么比虚拟机更快

    • 1)docker有着比虚拟机更少的抽象层
      由于docker不需要Hypervisor(虚拟机)实现硬件资源虚拟化,运行在docker容器上的程序直接使用的都是实际物理机的硬件资源。因此在CPU、内存利用率上docker将会在效率上有明显优势。
    • 2)docker利用的是宿主机的内核,而不需要加载操作系统OS内核当新建一个容器时,docker不需要和虚拟机一样重新加载一个操作系统内核。进而避免引寻、加载操作系统内核返回等比较费时费贪源的过程,当新建一个虚拟机时,虚拟机软件需要加载OS,返回新建过程是分钟级别的。而docker由于直接利用宿主机的操作系统,则省略了返回过程,因此新建一个docker容器只需要几秒钟。

  • 容器命令

  • 下载ubuntu镜像 docker pull ubuntu

  • 新建启动容器 docker run [option] image

  • option选项说明一个减号或者两个减号

    • –name= “” 为容器指定一个新名字 不指定的话 随机一个名字
    • -d 后台运行容器并且返回容器的id 就是启动守护式容器（后台运行）
    • -i 以交互式运行容器 通常和-t一起使用
    • -t 为容器重新分配一个伪输入终端 通常与-i同时使用 也即启动交互式容器(前台有伪终端，等待交互);
    • -P:随机端口映射，大写P
    • -p:指定端口映射，小写p **比如我们在容器的里面启动 redis 容器端口是6379 然后我们指定映射 -p 80 : 6379 此时我们只要访问宿主机的80 端口 就可以请求到我们docker容器里的redis **

  • 启动交互式容器 -it

如果想要退出的话使用 exit

  • 列出当前所有正在运行的容器 docker ps [options]
    • option说明
      • -a l列出当前正在运行的 和 历史运行的
      • -l 显示最近创建的容器
        • -n 显示最近的个创建的容器
      • -a 静默模式 只显示容器的编号

  • 启动守守护式容器 使用-d参数 docker run -d ubuntu 但是还有一个问题

当我们这样启动以后 我们再查看运行的情况 发现我们的容器并没有在运行

很重要的要说明的一点: Docker容器后台运行,就必须有一个前台进程容器运行的命令如果不是那些一直挂起的命令(比如运行top，tail)，就是会自动退出的。 所以我们刚-d参数启动容器 他就退出了’

这个是docker的机制问题,比如你的web容器,我们以nginx为例，正常情况下我们配置启动服务只需要启动响应的service即可。例如service nginx start但是,这样做,nginx为后台进程模式运行,就导致docker前台没有运行的应用这样的容器后台启动后,会立即自杀因为他觉得他没事可做了.所以，最佳的解决方案是,将你要运行的程序以前台进程的形式运行，常见就是命令行模式，表示我还有交互操作，别中断，

  • 现在有两种的启动方式 前台交互式启动 但是这样的有一个缺点 当我们的这个窗口关闭 redis就停止了

  • 后台守护式启动 docker run -d redis 启动守护式容器

这次没有自杀关闭是因为 redis本身启动就是一个前台程序一直都有进程 但是web启动都是后台进程 所以会自杀例如nginx

  • 查看容器启动的日志 docker logs 以后会讲到图形化的日志显示

  • 查看容器内进程的情况

  • 查看容器的内部细节

  • 进入正在运行的容器并以命令行交互 docker exec -it 容器id bashshell

  • docker attach id都能重新进入容器

上面两者的区别

attach直接进入容器启动命令的终端，不会启动新的进程，用exit退出，会导致容器的停止。
exec是在容器中打开新的终端，并且可以启动新的进程，用exit退出，不会导致容器的停止。

说白了就是exec就是新建一个新的终端 但是attach是使用正在运行的终端这样我们的attach的exit退出以后 这个进程就没了 就会自杀关闭了
